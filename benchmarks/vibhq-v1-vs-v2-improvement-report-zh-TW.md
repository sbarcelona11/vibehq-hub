# VibHQ 框架改進報告：V1 → V2

> 相同多智能體任務的控制對照實驗——「分析 $NVDA 並建立互動式 HTML 儀表板」——框架變更前後的比較。兩次執行使用相同的團隊組成（協調者 + 3 名工作者）、相同的使用者指令、相同的交付物規格。

---

## 基準測試任務

基準測試由五個循序階段組成，每個階段建立在前一個之上：

1. **研究** — 平行執行 NVDA 基本面與情緒分析
2. **資料包** — 結構化資料擷取，供儀表板使用
3. **靜態 HTML** — 精美單檔視覺化儀表板
4. **QA 驗證** — 資料準確性交叉驗證
5. **互動式 HTML** — 完整互動式儀表板，含滑桿、篩選器、圖表與資料匯出

此任務涵蓋所有協調機制：平行分派、循序依賴、產出物交接、資料 Schema 一致性、錯誤恢復與品質保證。

## 原生 LLM 行為問題發現

在 V1 測試期間，我們識別出七種當 LLM 在缺乏明確結構約束的多智能體協調中運作時會出現的湧現行為模式。這些並非模型本身的 Bug——它們是 LLM 處理指令方式的自然結果，且在不同模型系列（GPT-5.3 Codex、Claude Opus 4.6）中均可靠地重現。V2 中的每項框架變更都直接針對其中一個或多個行為模式。

### 1. LLM 在缺乏約束時會自由詮釋輸出格式

當任務指示「交付一個 JSON 資料包」時，每個 Agent 會獨立決定 Schema、key 命名慣例和結構。在 V1 中，三個 Agent 為同一個消費者產出了三種不相容的資料格式：

- Sarah 使用扁平頂層 key：`{ "meta": {}, "kpis": {} }`
- Mike 使用命名空間點記法：`{ "nvda.sentiment.composite": 6.2 }`
- Dave 自行建立了一份副本，而非重用 Sarah 的版本

這不是幻覺或誤解——每個 Agent 各自產出了合理的輸出。問題在於，缺乏共享合約時「合理」本身就是模糊的。

**框架修復：** 任務描述中的 `output_target` 和 `REQUIRED INPUTS` 欄位。協調者指定確切的檔名、預期格式，以及應消費而非重建的現有檔案。

### 2. LLM 急於立即開始工作

當 Agent 收到標記為 `QUEUED — 等待依賴` 的任務時，它會讀取任務描述並利用手邊已有的資訊開始工作。在 V2 測試中，Alan 收到一個附帶完整描述的排隊儀表板任務，立即接受並開始使用寫死資料建構，而非等待 Sarah 和 Mike 的 JSON 交付物。

這是 LLM 版本的「開發者看到 ticket 標題就開始寫 code，不等需求文件定稿」。模型的指令遵循偏差驅使它立即產出成果，而非閒置等待。

**框架修復：** QUEUED 任務現在隱藏完整任務描述。Agent 只會收到「存在一個任務，待依賴就緒後發送」的通知。完整描述——包括輸入檔案引用——僅在 READY 狀態時才交付。

### 3. LLM 傾向複製工作而非重用現有產出物

當被指派「用 NVDA 資料建構互動式儀表板」時，Dave 從零開始建立了自己的 `nvda-dashboard-data.json`，而非消費 Sarah 已發布的版本。模型的預設行為是在自己的 context 中生成所需的一切，因為從記憶中生成是阻力最小的路徑，而讀取外部檔案則需要工具呼叫。

**框架修復：** `REQUIRED INPUTS（不要自行重建）`區塊搭配明確的檔案引用。指令「讀取並遵循：nvda-dashboard-fundamentals-pack.md（由 Sarah 擁有）。不要建立自己的版本」直接對抗了「生成優先於檢索」的偏差。

### 4. LLM 使用指標引用而非完整內容

當 Agent 呼叫 `publish_artifact` 時，它們經常寫入一個引用字串，如 `"See shared file nvda-fundamentals-data.json"`（43 bytes），而非嵌入完整的 10KB 內容。Agent 已透過 `share_file` 寫入完整內容，並將 `publish_artifact` 視為公告而非資料傳輸——這是一個合理的詮釋，但會破壞直接讀取 artifact 內容的下游消費者。

這是跨模型的持續性模式：當 LLM 已經「完成」工作（透過 `share_file`），它會透過摘要而非重複來優化後續呼叫。Stub 問題消耗了 V2 協調者 21% 的訊息和 68 分鐘的執行時間。

**框架修復：** Hub 端內容驗證，拒絕 `publish_artifact` 中 `content.length < 200` 的呼叫，強制 Agent 包含實際內容。此外，`share_file` 現在會自動註冊 artifact 條目，完全消除對單獨 `publish_artifact` 呼叫的需求。

### 5. LLM 在壓力下偏離指派角色

協調者（Emma）被設計為專職任務協調——建立任務、監控狀態、路由產出物。但當 V1 遇到 Sarah 和 Mike 資料間的 Schema 衝突時，Emma 轉而撰寫 JavaScript 轉接器程式碼。當瀏覽器封鎖 `file://` XHR 載入時，Emma 撰寫了 JS 資料 fallback。一個絕不應該寫程式碼的 Agent 進行了六次獨立的程式碼介入。

這是自然的 LLM 行為：當面對問題且配備了程式碼工具時，模型會直接解決而非委派。「協調者」角色是系統提示中的軟約束，容易被模型的問題解決驅力覆蓋。

**框架修復：** 結構化合約消除了大多數觸發角色飄移的整合失敗。剩餘的緩解措施是協調者系統提示規則：「不要撰寫實作程式碼——為工作者建立新任務」。這部分有效，但當模型感知到緊迫性時可能被覆蓋（例如所有工作者均無回應）。

### 6. LLM 無法有效自我驗證

一個同時生成分析資料並用該資料建構儀表板的單一 Agent，對準確性沒有獨立檢查機制。在 V1 中，研究備忘錄中的資料錯誤未經檢查地傳播到最終儀表板，因為同一概念「管線」既生產又消費了資料。

這是一個根本性限制：LLM 對自身先前的輸出展現確認偏差。當被要求「驗證」自己生成的資料時，它們傾向於確認而非挑戰。

**框架修復：** 專屬 QA 階段，由獨立 Agent（Sarah、Mike）對照來源文件驗證其他 Agent 產出的資料。V2 的 QA 階段在 67 個驗證項目中攔截了 7 個資料準確性問題——這些錯誤在 V1 中會未經檢查地到達最終儀表板。

### 7. LLM 不會主動發出故障信號

當 Agent 的 CLI session 當機、WebSocket 斷線、或模型觸發速率限制時，不會發送任何故障信號。協調者繼續分派訊息並等待永遠不會到來的回應。在 V2 測試（獨立執行）中，這導致了 18 分鐘的閒置等待，之後協調者才觸發手動 fallback。

LLM 沒有「我即將變得不可用」的內建概念。它們只是停止回應，而這種沉默與執行緩慢的任務無法區分。

**框架修復：** Agent 心跳監控。Hub 追蹤最後活動時間戳，並在可設定的 timeout 後將 Agent 標記為 `agent_unresponsive`。協調者會收到自動通知，可以立即重新分派受阻的任務。

## 各階段詳細比較

### 階段一：平行研究

兩個版本皆同時分派兩個獨立研究任務。

```
Emma ──create_task──▶ Sarah:「NVDA 基本面與估值分析」
     ──create_task──▶ Mike: 「NVDA 情緒與市場定位分析」
```

| | V1 | V2 |
|---|---|---|
| Sarah 交付時間 | 3 分鐘 | 5 分鐘 |
| Mike 交付時間 | 4 分鐘 | 5 分鐘 |
| 產出格式 | Markdown 備忘錄 | Markdown 備忘錄 |
| 問題 | 無 | 無 |

**評估：表現相當。** 完全平行的研究任務是最簡單的協調模式，兩個版本都能良好處理，因為任務完全獨立、無共享狀態。

### 階段二：儀表板資料包

協調者請求結構化資料擷取，用於儀表板生成。

| | V1 | V2 |
|---|---|---|
| 交付時間 | ~2 分鐘 | ~2 分鐘 |
| 產出規格 | 僅自然語言描述 | 明確的 `📤 EXPECTED OUTPUT` 含檔名 |
| 產出格式控制 | 無——Agent 自行決定 | `output_target` 欄位指定檔名和位置 |
| 問題 | 當下無顯現 | Mike 短暫無回應 → Emma 自動將備份任務分派給 Sarah |

**V2 改善：`output_target` 欄位。** V1 的任務以純自然語言描述，將產出格式留給各 Agent 自行解讀。V2 任務包含機器可讀的產出規格：

```
V1 任務描述（節錄）：
  "將你的 NVDA 分析濃縮為儀表板就緒的資料包..."

V2 任務描述（節錄）：
  "⚠️ 輸出目標：
   建立檔案：nvda-dashboard-fundamentals-pack.md
   📤 預期產出：
   發布 artifact：nvda-dashboard-fundamentals-pack.md"
```

此差異在階段五（多個 Agent 必須產出相容資料）時變得至關重要。

### 階段三：靜態 HTML 儀表板

| | V1 | V2 |
|---|---|---|
| 建構者 | Emma（獨自完成） | Alan（專職前端 Agent） |
| 時間 | 10 分鐘 | 7 分鐘 |
| 方式 | 協調者直接生成 HTML | 透過任務委派，明確列出輸入檔案 |
| 資料來源 | 從記憶中寫死 | 透過 `read_shared_file` 讀取共享檔案 |
| 問題 | 此階段無 | `publish_artifact` 回傳佔位文字 → Emma 偵測後要求透過 `share_file` 重新上傳 |

**V2 改善：輸入檔案綁定。** V2 任務包含 `📥 必要輸入` 區塊，明確列出 Agent 必須使用的檔案：

```
📥 必要輸入（不要自行重建）：
- 讀取並遵循：nvda-detailed-dashboard-V2.md（由 Emma 擁有）
- 讀取並遵循：nvda-dashboard-fundamentals-pack.md（由 Sarah 擁有）
- 讀取並遵循：nvda-dashboard-sentiment-pack.md（由 Mike 擁有）
```

這防止了 V1 階段五中出現的「每個人各做各的」失敗模式。

**V2 的已知問題：Stub 偵測。** `publish_artifact` 工具仍接受了 143 bytes 的佔位字串。但 Emma 在 2 個 turn 內偵測到此問題（「copied file looks suspiciously small... placeholder string」），並透過要求 `share_file` 重新上傳完整內容（69KB）恢復。V1 中此失敗模式不存在，因為檔案透過本地檔案系統交換。

### 階段四：QA 驗證（僅 V2）

V2 引入了 V1 完全缺乏的正式交叉驗證步驟。

```
Emma ──create_task──▶ Sarah:「基本面 QA 包」
     ──create_task──▶ Mike: 「情緒 QA 包」
```

**Sarah 的 QA 結果**（驗證 67 個項目）：
- 60 項通過
- 7 項標記並修正：
  - 預期報酬率修正為目標價 `$177.80` 的 `+18.8%`
  - 遊戲部門 QoQ 數據修正為 `-14.0%`
  - EPS 數據標註 GAAP 與非 GAAP 區別
  - 提供修正後的表格供直接合併至儀表板

**Mike 的 QA 結果**：
- 12/12 催化事件日期經來源文件驗證
- 6/6 選擇權指標確認（買賣權比率、隱含波動率、偏斜度）
- 觸發機率情境驗證：所有情境總和等於 100%
- 紅綠燈狀態指標確認一致

**評估：全新功能。** QA 階段攔截了 7 個資料準確性問題，這些問題在 V1 中會未經檢查地到達最終儀表板。這本質上是多智能體才有的能力——獨立的 Agent 從頭審查另一個 Agent 的產出，提供單一 Agent 無法對自身輸出複製的獨立驗證。

### 階段五：互動式儀表板——關鍵測試

此階段是 V1 災難性失敗而 V2 成功之處。它需要最緊密的協調：多個 Agent 必須產出格式相容的資料，由前端 Agent 消費並建構統一的互動式應用。

#### V1：三個 Agent，三種資料模型，一個損壞的儀表板

**發生了什麼：**

Sarah、Mike 和 Dave 各自獨立產出資料，沒有共享的 Schema 契約：

```
Sarah 的 JSON：
  { "meta": {...}, "kpis": {...}, "valuation": {...} }
  → 頂層 key，扁平結構

Mike 的互動規則：
  { "nvda.sentiment.composite": 6.2, "nvda.gammaZones": [...] }
  → 命名空間點記法，巢狀陣列

Dave 的 JSON（另外建立）：
  { "meta": {...}, "financials": {...} }
  → 不同的頂層 key，從 Sarah 複製的重複資料
```

**失敗連鎖反應：**

1. **偵測到 Schema 衝突**（13:50）— Emma 發現 Mike 的命名空間路徑與 Sarah 的扁平結構不相容
2. **協調者角色飄移**（13:53）— Emma 撰寫 JavaScript Schema 轉接器來橋接格式，將協調者 turn 花在撰寫實作程式碼上
3. **轉接器 Bug**（13:53）— 轉接器在 Sarah 的 `meta` key 上短路，需要第二次修補
4. **瀏覽器載入失敗**（13:56）— 儀表板透過 `XMLHttpRequest` 載入 JSON，瀏覽器在 `file://` 協議下封鎖此操作
5. **緊急 JS Fallback**（13:57）— Emma 建立內嵌 JavaScript 資料 fallback 來繞過 XHR 問題

**最終結果：** 一個脆弱的儀表板，需要協調者的 3 輪手動程式碼修補，記錄了 15 次 Schema 相關事件和 6 次手動程式碼介入。使用者首次開啟時回報「Failed to load dashboard data — Network error」。

#### V2：契約優先，零 Schema 衝突，正常運作的儀表板

**發生了什麼：**

框架變更確保所有 Agent 依循共享契約工作：

1. **先定義結構化 JSON 模型** — Sarah 和 Mike 各自交付了正式的資料模型文件（`nvda-dashboard-fundamentals-data-model.md` 和 `nvda-dashboard-sentiment-data-model.md`），在 Alan 開始建構之前提供明確的 Schema 定義
2. **Alan 在建構前讀取所有輸入** — 任務描述中的 `📥 必要輸入` 區塊列出了他必須使用的每個檔案，Claude Code 的 `read_shared_file` 工具擷取了實際內容
3. **單檔交付** — Alan 產出了自包含的 62KB HTML，所有資料內嵌（無外部 JSON 載入），徹底消除了 `file://` 協議問題

**Schema 事件：** 2 次（V1 為 15 次）— 兩次都是次要的命名澄清，非結構性不相容。

**協調者手動程式碼介入：** 0 次（V1 為 6 次）。

**最終結果：** 一個正常運作的 62KB 互動式儀表板，包含全部 8 項要求的功能：
1. 動畫分頁導航（6 個分頁含過渡動畫）
2. 時間範圍切換（季度 / 週度檢視）
3. 情境機率滑桿與即時加權目標價
4. 情緒量表含組成分解
5. 可篩選的催化事件時間線
6. 可排序/搜尋的資料表格
7. 觸發機率模擬器
8. JSON 資料匯出

## 框架變更及其實測影響

### 變更一：結構化任務描述與 `output_target`

**解決的問題：** V1 中任務描述為純自然語言，Agent 各自解讀產出格式、檔名和位置，導致不相容的交付物。

**實作方式：**

```
V2 任務建立包含：

⚠️ 輸出目標：
建立檔案：nvda-dashboard-fundamentals-pack.md

📥 必要輸入（不要自行重建）：
- 讀取：nvda-fundamental-memo.md（由 Sarah 擁有）
- 讀取：nvda-sentiment-memo.md（由 Mike 擁有）

📤 預期產出：
發布 artifact：nvda-dashboard-fundamentals-pack.md
建立共享檔案：nvda-dashboard-fundamentals-pack.md
```

**實測影響：**

| 指標 | V1 | V2 |
|---|---|---|
| Schema 衝突 | 15 次事件 | 2 次事件 |
| Agent 重建輸入檔案 | 有（Dave 自建 JSON） | 無 |
| 協調者撰寫轉接器程式碼 | 有（6 次介入） | 無 |
| 首次嘗試即相容交接 | 否 | 是 |

### 變更二：Agent 健康監控與心跳機制

**解決的問題：** V1 中當 Agent 的 CLI session 當機或斷線時，協調者無法偵測，持續向無回應的 Agent 發送訊息。

**實作方式：** 透過活動時間戳監控 Agent。若在 timeout 視窗內未收到回應，Agent 被標記為 `agent_unresponsive`，協調者自動收到通知。

**實測影響：**

| 指標 | V1 | V2 |
|---|---|---|
| 離線 Agent 偵測 | 從未偵測 | ~60 秒內偵測 |
| 等待無回應 Agent 浪費的時間 | 18 分鐘以上（V2 測試中） | ~0 分鐘（立即重新分派） |
| 自動任務重新分派 | 無 | 有 — Emma 將 Mike 的受阻任務重新分派給 Sarah |
| 冗餘 Fallback 工作 | N/A | 輕微 — Emma 在 Alan 交付前短暫建了自己的 HTML |

**已知限制：** 目前 timeout（~60 秒）可能過於激進。在 V2 執行中，Alan 和 Mike 在長時間執行任務（HTML 生成、批次 web search）期間被短暫標記為無回應，但隨後成功完成。這導致 Emma 開始了冗餘的 fallback 工作。建議調整：將 timeout 增加至 3-5 分鐘。

### 變更三：產出物所有權與檔案共享

**解決的問題：** V1 中多個 Agent 可以寫入相同檔名，導致靜默覆寫。沒有集中式的產出物登錄。

**實作方式：** 每次 `publish_artifact` 呼叫都會登記所有權。協調者可查詢 artifact metadata，查看每個檔案由哪個 Agent 在何時產出。

**實測影響：**

| 指標 | V1 | V2 |
|---|---|---|
| 檔案覆寫 | 可能（無保護） | 0（追蹤所有權） |
| 協調者知道 artifact 來源 | 否 | 是 — 日誌顯示擁有者、時間戳、大小 |
| 重複工作（多個 Agent 產出相同檔案） | 有（Dave 重建了 Sarah 的 JSON） | 無 |

**已知限制：** `publish_artifact` 工具仍接受佔位內容（例如「See local file: nvda-analysis-dashboard.html」— 143 bytes）。`share_file` 工具能正確傳輸完整內容（69KB），但 Agent 同時呼叫兩者，造成混淆的雙軌制。計劃修復：Hub 端驗證拒絕 `publish_artifact` 中 `content.length < 200` 的呼叫。

### 變更四：QA 驗證階段

**解決的問題：** V1 沒有在最終交付物前交叉檢查資料準確性的機制。研究備忘錄中的錯誤會未經檢查地傳播到儀表板。

**實作方式：** 資料包交付後，協調者分派 QA 任務給獨立 Agent，由其對照來源文件驗證數據。

**實測影響：**

| 指標 | V1 | V2 |
|---|---|---|
| 最終產出前的資料準確性檢查 | 0 | 67 項驗證 |
| 儀表板前攔截的錯誤 | 0 | 7 項修正 |
| 最終儀表板中的錯誤 | 未知 | 0 已確認 |

這是唯一本質上需要多智能體協調的功能——獨立 Agent 審查另一個 Agent 的工作，提供單一 Agent 無法對自身產出複製的獨立驗證。

## 失敗恢復比較

### V1 失敗恢復：協調者變成開發者

V1 遇到 Schema 衝突時，協調者（Emma）從協調角色轉為實作角色：

```
時間線：
13:50  Emma 偵測到 Sarah 和 Mike 資料間的 Schema 衝突
13:53  Emma 撰寫 JavaScript Schema 轉接器（第一次嘗試）
13:53  轉接器有 Bug — 在 'meta' key 上短路
13:55  Emma 修補轉接器（第二次嘗試）
13:56  使用者回報：「Failed to load dashboard data — Network error」
13:57  Emma 建立 JS 資料 fallback 檔案
13:59  Emma 修補 HTML 載入器以使用內嵌 JS 資料
```

由一個設計為任務協調的 Agent 進行了**四輪手動程式碼修補**。每次修補都在協調者的對話 context 中加入程式碼，增加了後續所有操作的每 turn 成本。

### V2 失敗恢復：偵測、重新分派、繼續

V2 遇到問題時，框架的內建機制處理了它們：

```
時間線：
05:49  Emma 偵測到 Mike 為 agent_unresponsive
05:49  Emma 建立備份任務，分派給 Sarah（11 秒）
05:49  Mike 交付原始任務（只是比較慢，沒有當機）
05:50  Emma 將備份標記為已取代，使用 Mike 的產出
       → 協調者零程式碼撰寫、零手動修補

06:17  Emma 偵測到 Alan 的 publish_artifact 是佔位文字（143 bytes）
06:17  Emma 要求透過 share_file 重新分享
06:24  Alan 以完整 69KB 內容重新分享
06:24  Emma 同步至工作區，驗證正確大小
       → 2 個 turn 解決、零程式碼修補
```

**零次手動程式碼介入。** 每個失敗都透過協調協議（重新分派、重新請求、驗證）解決，而非協調者進入開發者角色。

## 已知限制與計劃改善

| 問題 | 現狀 | 計劃修復 |

## 結論

V1 → V2 的框架變更解決了 V1 失敗的三個根本原因：

1. **Schema 衝突 → 已消除。** 結構化任務描述含 `output_target`、`必要輸入`、`預期產出` 欄位，確保所有 Agent 依循共享契約工作。Schema 衝突事件從 15 次降至 2 次（-87%）。

2. **協調者角色飄移 → 已消除。** 當失敗發生時，框架的協調機制（重新分派、重新請求、驗證）處理恢復，協調者不需要撰寫程式碼。手動程式碼介入從 6 次降至 0 次。

3. **靜默失敗 → 可偵測。** Agent 健康監控在 ~60 秒內偵測到無回應的 Agent。Artifact metadata 追蹤來源和大小，實現 stub 偵測。兩者都是 V1 完全缺乏的新功能。

最終結果：**端到端加速 46%**、**正常運作的交付物**（相較 V1 損壞的儀表板）、以及新 QA 階段**攔截了 7 個資料準確性問題**——同時維持相當的整體複雜度。

框架的協調開銷仍是主要成本因素，對於小範圍任務，單一 Agent 顯著優於多智能體協調。多智能體的價值主張出現在需要獨立驗證（QA）、異質工具存取、或超過單一 Agent context window 容量的工作負載中。未來以更大範圍專案進行的基準測試將進一步驗證交叉點的位置。
